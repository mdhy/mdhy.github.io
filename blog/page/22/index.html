帰省二日目&三日目

Sep 13th, 2008

まだ「蟲師」を観ています。

予てから計画していた実家のPCを遠隔操作するための構築が完成しました♪

それはopenVPNとTightVNCの合わせ技です。

低速なADSL回線でも実用的です。

いま、東京の自宅にあるWindows端末にリモートデスクトップで接続して、その端末はOpenVPNで構築した仮想ネットワークから実家のPCをTightVNCで遠隔操作していますが、実用の範囲です。

この構成は端末が起動するとOpenVPNが接続してOpenVPNサーバに接続してくるので、動的グローバルアドレスであっても仮想ネットワーク上のアドレスは変わらないため、端末が起動していればVNCで接続することができます。

また、TightVNCが優秀なのでオーバーヘッドの大きくなるVPNでも接続先の端末が低速なADSLでつながっていることを考えれば、十分実用的に使えていますという話。

OpenVPNとTightVNC

Sep 12th, 2008

実家に帰ってもネットワーク構築を止めない林です。

予てから妄想していた実家のPCをリモコン（リモートコントロール）する環境の構築がすんなり出来上がりました。

＝＝＝＝＝＝

実家のPCはWNDOWSでありWINDOWSをリモコンする方法として実現する方法として、以下が考えられる。

・リモートアシスタント（動作が軽快そうだが、接続手続きが複雑）

・VNC（動作が鈍重になりやすい。MacbookでViewerを使うと轟音をあげる）

・LogInMe（候補だけど面白みが無い。フリーで実現できることが限られる）

・リモートデスクトップ（接続先の画面と操作を奪ってしまうので、論外だがMacbookで使ってもFANが轟音をあげない）

＝＝＝＝＝＝

リモートアシスタントが一番やりたかったことなんだけど、手続きを簡略かするためには、ドメイン参加をさせなければいけないらしい。

しかもリモートアシスタンス関連の設定（ドメインADサーバ）でグループポリシーが使えなければいけないらしいので、必然的にWindows Server 2003が必要になります。

SAMBA4系からグループポリシーに対応するということなので、アルファ版を試しに使ってみたけど、仮想ファイルシステムの辺でコケて間に合いそうに無いので、ドメイン構築をあきらめました＞＜

自動的にリモートアシスタンスを使うことも延期になりました。

＝＝＝＝＝＝

ドメインの構築を行うためにネットワーク的な部分も実現に向けてアプローチする必要がありました。

インターネット上に存在する２拠点をドメインに参加させる必要がありました。

該当の２拠点は市販ブロードバンドルータのNAT下にあり、グローバルアドレスは動的です。

そなるとWINSサーバとルーティングではうまくいきませんし、流れるデータを暗号化する必要があります。

そうなると考えつくところはVPN（仮想プライベートネットワーク）になるわけです。

アプローチとしては、１拠点の端末ごとにVPNの設定をすると面倒くさいという理由からブローバンドルータのNAT下のすべての端末をVPNを介して通信できるようにネットワークをSSL-VPNで構築しようと奮起しましたが、機器を使った方が早いんですが、今後拠点が増減することと構築の重要性から投資は無意味だと判断してあきらめました。

＝＝＝＝＝

完全に行き詰まったのでググってみるとOpenVPNの存在を知る訳です。

OpenVPNは十数行の設定ファイルとルート証明書・クライアント（サーバ）証明書・TLS/SSL証明書でクライアントを構築できます。（サーバも同様）

OpenVPN Japanese Infomation : http://freescitech.net/2/wiki/

SAMBAの共有プリンタでPDFをつくる

Aug 27th, 2008

さいきん気温の変化でメタボ肌になっている林です。

さて、「SAMBAでPDFを・・・」なんてネタはもう古いです。

２００４年頃に集中して備忘録らしいものが複数Googleで確認できますー

なんでも自分で再現して検証しないと気がすまないので、ちょっくら部屋の中でファイル共有に使うために運用し始めたものがあるので設定を追加してみました。

参考URL１：http://www.takamisawa.org/smbpdf/index.html

参考URL２：http://furyo.on-air.ne.jp/linux/pdf_server.html

二つほど参考にしたサイトを紹介しましたが、そのとおりにしたら動きませんでした＞＜

動かなかった原因は、

cupsを使わないのにcupsにデータを飛ばしていた。

そう、sambaをパッケージでインストールしてしまった場合はcupsにデータを飛ばすようです＞＜

ならソースからコンパイルするしか？？と思ったのですが、

[global]

・・・

printing = bsd

[global]に「printing = bsd」を追加することで、デフォルトcupsを解除できます。

注意ですが、上記の設定を行った場合は、「print command =」 でどのコマンドでどの引数で実行するかを明記する必要があります。

なので任意に実行するプログラムを指定できるようになるわけです♪

これだけではどうしてなのか、うごきませんでした；；

そこで[printers]という項目にオプションを一つも付けずに書くと動くようになります♪

インターネットで得た資料は「動かないんですけど？」→「こうしたら動きますよ」的なのもや備忘録的なものはあるんですが、前提条件が書かれていないので自身が構築した環境と比べることができないのがハマりやすい原因のようです。

こうなったら、ログを解析して、それっぽいところを検索して、オフィシャルのドキュメントを読んでとかになるんですよ＞＜

＝＝＝＝＝＝

というわけで、再現ができたので満足です。

今回構築した環境は

CentOS5.1

samba3.x パッケージ

今回役に立ったコマンドを紹介すると

testparm -v

読み込まれているオプションとその値を確認することができます。

注意ですが、オプションが明示的に指定しないとデフォルト値が入れられます。なので、何も持たせたくない場合は「option = 」と書いておいた方が無難です。

いまさらsamba(サンバ)の備忘録でしたー
