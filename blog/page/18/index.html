正月帰省４日目

Jan 3rd, 2009

１日遅れでお送りしています。

８０人くらい集まって、１次会？は近況報告したり、写真撮ったり、とりあえず食ったり飲んだりしてワイワイしてました。

私はというと、当時仲の良かった？人に絡んでまわって、とりあえず「いま何してるの？どこ住んでるの？」の質問攻めでした。

どうも東京に住んでいて同業種に就職している人も多くいて５人くらいと連絡先を交換しました。

もう久しぶりにテンション上がっちゃって、２次会もガツガツ酒飲んで１人で大声で騒いでいたような気がします。

車で来たのに、厚かましく家の近い友達に帰りに相乗りさせてもらえないか頼んで回って、結局まったく違う方角の友達に送ってもらいました。

ほんとありがとうございました。

もちろん、けさ２０２０円を出してタクシーで車を回収するはめになりました。

＝＝＝＝＝＝

話変わって、厄年の話です。

数え年で２５歳（男性）が「小厄」の本厄らしいです。

ということは私の場合、平成１９年（前厄）・平成２０年（本厄）・平成２１年（後厄）ということになります。

お守りは貰いましたが、厄払いなど考えたこともなく、１年間生活していました。

本厄の年には、引越や新たなチャレンジはしないのが無難だそうですが、私の場合は東京に引越をして，就職しましたｗ

確かに去年はやる気は最悪でしたが、私のバイオリズムは１年目「黙」、２年目「暴」、３年目「落」をくり返しているので、そんなもんだろうなんて思っていましたが、なんかあったみたいです。

今年も後厄らしいので、静かに生きていきます。

もしかしたら私の厄を受けた人がいるとしたら、ごめんなさい。

返そうとしても、受け取りはしませんが＞＜

＝＝＝＝＝＝

あといまさら、去年を漢字１文字で表すと

「黙」

です。

今年の目標を漢字１文字で表すと

「静」

にしようと思っています。後厄なので

MySQLに移行完了しました

Dec 23rd, 2008

日曜日にスゴい風でまったく寝付けずに昨日は１２時間くらい寝てしまった林です。

睡眠は人生を無駄にしていると言われますが、たしかにそのとおりでしょうｗ

＝＝＝＝＝＝

このブログのデータ（PostgreSQLに格納されている）をMySQLに移すことが簡単に実現できたことを以前の記事で紹介しました。

以前の記事：

次はこのブログのデータベースとの接続部分をMySQLと通信できるように変えないといけません；；

ここが面倒くさいなーーーということなんですが、このブロクを最後に書いたのは今年の３月くらいでした。

たしか、PostgreSQLでもMySQLでもすぐにセットアップできるように設計したこと思い出して、データベース関連のソースを見てみると、それっぽく書いてありました。（書いたのは私自身なんですが；；）

過去の記憶を手探りしながら、一日コースだな＞＜なんて思いながら、手を加えてみると以下のようになりました。

function sql_connect($IN_USER=‘’,$IN_PASS=‘’,$IN_DBNAME=‘’,$IN_HOST=‘localhost’,$IN_PORT=‘5432’){

$db_my = new PDO(“mysql:host=$IN_HOST;dbname=$IN_DBNAME”,$IN_USER,$IN_PASS);

$db_my->query(‘SET NAMES utf8’);

return $db_my;

// return pg_connect(“user=$IN_USER password=$IN_PASS dbname=$IN_DBNAME host=$IN_HOST port=$IN_PORT”);

}

function sql_close($IN_CON){

$IN_CON = null;

return true;

// return pg_close($IN_CON);

}

function sql_query($IN_CON, $IN_STRING){

return $IN_CON->query($IN_STRING);

// return pg_query($IN_CON, $IN_STRING);

}

function sql_fetch_row($IN_OBJECT){

return $IN_OBJECT->fetch(PDO::FETCH_BOTH);

// return pg_fetch_row($IN_OBJECT);

}

function sql_fetch_array($IN_OBJECT){

return $IN_OBJECT->fetchall(PDO::FETCH_BOTH);

// return pg_fetch_all($IN_OBJECT);

}

function sql_num_rows($IN_OBJECT){

return $IN_OBJECT->rowCount();

// return pg_num_rows($IN_OBJECT);

}

だけ直せば問題ないようでした。

ただし、SELECT文でカラム名を大文字で指定するとPDO::FETCH_BOTHの配列の結果も大文字で返ってくるため、不具合がでるところも数カ所ありました。

が、小文字で指定してやれば小文字で返ってきたので問題ありませんでした。

＝＝＝＝＝＝

これで、無事に移行ができそうです。

来年を予定してましたが、気持ちよく年を越せそうです♪

PHP:PDOってすごい幸せ

Dec 21st, 2008

１０代目サーバに移行するための準備をしています。

９代目サーバでは、MySQLサーバとPostgreSQLサーバを動かしていましたが、１０代目サーバからはMySQLサーバのみにすることにしました。

理由は、データベースの差分を取るのにそれぞれ違うサーバだと操作が違って面倒だからですｗ

それで、このブログのデータはPostgreSQLで格納しているのですが、MySQLにデータを移したいわけです。

方法としてphpでMySQLとPostgreSQLと通信する関数が用意されているので、それぞれに接続して、SELECT文で取り出してWHILE文とか使って、INSERT文で移し替えればいいわけです。

このブログのデータも以前はMySQLに収まっていたものをPostgreSQLに移し替えた経験からいうと、

正直、面倒くさい！！

あれから時間が経っているし、私自身のコピペの才能も成長したので、ググってみた。

＝＝＝＝＝＝

そして見つけたのは、PDOでした。

”PDO（PHP Data Object）は、PHP 5から利用できるデータアクセス抽象化レイヤです。”

というもので、PHPで用意されているMySQLやPostgreSQL専用の関数を使うことなく、データベースとの通信が扱えるというものらしい。

下準備として、PHPがPDOモジュールを読み込んでないと始まりません。

ソースからコンパイルするなら、

./configure —enable-pdo —with-pdo-mysql=/opt/mysql —with-pdo-pgsql=/opt/pgsql

ってなオプションをつけてコンパイルすると使えるようになります。

もっとまともな説明を知りたい方は、google先生に聞いてください。

＝＝＝＝＝＝

下準備ができたら、実際にコードを書いてみます。(MySQLの場合)

$db_my = new PDO(mysql:host=localhost;dbname=test”,“root”,“”);

これで接続できました。

もしエラーを制御したいなら

try {

$db_my = new PDO(mysql:host=localhost;dbname=test”,“root”,“”);

} catch(PDOException $e) {

echo $e->getMessage();

}

と書けば処理を中断させずにエラーメッセージを出力できます。

PostgreSQLなら

$db_my = new PDO(pgsql:host=localhost;dbname=pgsql”,“pgsql”,“”);

です。

＝＝＝＝＝＝

今回やりたいことは、PostgreSQLからMySQLにデータを移すことです。

要は同じテーブルを用意してコピーすればいんですが、PDOならこうすれば実現できるようです。

try {

// PDOでデータベースで接続

$db_my = new PDO(“mysql:host=localhost;dbname=myblog”,“root”,“”);

// 文字化け対策（原因はクライアントがlatin1なのが原因???）

$db_my->query(‘SET NAMES utf8’);

$db_pg = new PDO(“pgsql:host=localhost;dbname=pgblog”,“pgblog”,“”);

// ６行でテーブルコピーができる。幸せｗ

$rs = $db_pg->query(‘SELECT * FROM pg_index’);

$sth_my = $db_my->prepare(‘INSERT INTO my_index VALUES(?,?,?,?,?)’);

while($r=$rs->fetch(PDO::FETCH_NUM))

{

$sth_my->execute($r);

}

} catch(PDOException $e) {

print(‘エラーメッセージ：’.$e->getMessage());

}

＝＝＝＝＝＝

これが本来？のプログラミングの仕方なんでしょうねー

いやーほんと助かりましたよ♪

これで１０代目サーバに移行できる日は近いです。

自宅サーバを１０代目に移行中

Dec 21st, 2008

自宅の省エネ（電気代削減）を進めるために、９代目サーバ（３ヶ月運用）を１０代目に移行することにしました。

９代目からXenを使った仮想化でwinxpを動かして運用してきましたが、リモートデスクトップ（以下、RDP）を使ってごにょごにょしている分には、とくに不満はありませんでした。

しかし、現状ではCentOSとwinxpを動かすだけで、力一杯の感じなので、もう一つWINDOWS系サーバ（入手できるのでだとWHS。金銭的な問題でw）を動かして、実験用にもう一つなにかOSが動かせたらと欲張りなことを実現することができません。

現状の９代目サーバは

・CPU Intel Core2Duo E6600

・Memory 2GB (ECC 512MB x 4)

・HDD 80GB SATA

ってところです。

なら、筐体２台でいいんじゃない？というわけですが、グリーンITですから１台でいきたいんですよ♪（実際に何かをしている時間は少ないので、２台もいらないんです。）

そんなわけで１０代目に移行した理由は、９代目がDELLのSC440であることから、ECCメモリしか使えず、投資しても対費用効果が得られないため、HPのML115を使うこと（ML115はnon-ECCメモリが使える）を決めたからです。（自宅サーバなので、non-ECCでも、問題ないと思い込んでるw）

１０代目サーバは

・CPU AMD Phenom X4 9350e

・Memory 4GB (non-ECC 2GB x 4)

・HDD 80GB + 160GB + 80GB(追加予定)SATA ←SoftwareRAID(RAID1)+LVM

ってところで落ち着きました。

予算は・・・とくになく、とりあえず勢いで揃えてみると

・ML115 G512200円

・４GBメモリ4480円

・AMD Phenom X4 9350e13900円

くらいで、約３０５８０円を浪費してましたｗ

＝＝＝＝＝

これで本格的に仮想化を動かせれて、RAID0で安心となるわけです♪

ESXiとか様々な思案しましたが、CoentOS5 + Xenで今回は収まりました。

仮想化のメリット（Xenの場合）は、柔軟にCPU・メモリ資源を再配分できること・仮想ネットワークを構築できること、互いに負荷が独立していること等です。

デメリットは、物理的なデバイスを扱いにくいということです＞＜

まぁRDPから使ったり、サーバを動かしたりして使おうとする分には、デメリットはネックになりにくいですよ（私自身の経験から）

＝＝＝＝＝

んま、次はコンテンツ開発ですか（いつからいってるやら。とほほ）・・・開発は自宅サーバでもいいのですが、コンテンツの運用はホスティングにしますよ。。。ほんと＞＜

＝＝＝＝＝

ホスティングとかいってますが、実家にも光インターネットが来たので、活用しようと目論んでいます♪
